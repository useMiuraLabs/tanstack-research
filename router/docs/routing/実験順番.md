# TanStack Router 実験順番

TanStack Router のファイルベースルーティングを段階的に体感するための実験計画です。Next.js (App Router) との対比を添えながら、ファイル名規約・型推論・レイアウトの仕組みを順を追って学べる構成にしています。

## 全体像

| #   | 実験名                        | 学ぶ概念                    | Next.js 対応                          |
| --- | ----------------------------- | --------------------------- | ------------------------------------- |
| 1   | Index と Static Route         | ファイル名 = URL の最小単位 | `app/page.tsx` / `app/about/page.tsx` |
| 2   | Dynamic Route                 | 型安全な params             | `app/posts/[postId]/page.tsx`         |
| 3   | Layout Route と `<Outlet />`  | レイアウト共有              | `app/posts/layout.tsx`                |
| 4   | Pathless Layout (`_auth.tsx`) | URL に出ない layout         | `app/(auth)/...`                      |

---

## 実験 1: Index と Static Route を体感する

### ファイル例

- `src/routes/__root.tsx` (既存)
- `src/routes/index.tsx` (既存)
- `src/routes/about.tsx` (新規追加)

### 狙い

- ファイルを保存した瞬間に `src/routeTree.gen.ts` が自動更新される様子を観察します。
- `<Link to="/about">` を書いた際に、`to` の補完候補に `/about` が現れることを確認します (型レベルでルートが効いている感覚を掴みます)。

### 確認手順

1. `pnpm dev` (もしくは `npm run dev`) で開発サーバーを起動します。
2. `src/routes/about.tsx` を作成し、`createFileRoute('/about')` で最小の component を返します。
3. エディタで `routeTree.gen.ts` を開きながら保存し、差分が自動生成されることを確認します。
4. `__root.tsx` のナビゲーションに `<Link to="/about">` を追加し、補完が効くこと、クリックで遷移できることを確認します。

### Next.js 対比

| 観点       | TanStack Router                          | Next.js (App Router)             |
| ---------- | ---------------------------------------- | -------------------------------- |
| ファイル   | `src/routes/about.tsx`                   | `app/about/page.tsx`             |
| ルート定義 | `createFileRoute('/about')` で明示       | ファイル配置のみで暗黙的         |
| 型補完     | `<Link to>` がユニオン型として補完される | `<Link href>` は `string` のまま |

## 実験 2: Dynamic Route で型の力を見る

### ファイル例

- `src/routes/posts.$postId.tsx`

### 狙い

- ファイル名の `$postId` がそのまま params のキーになり、`Route.useParams()` が `{ postId: string }` として型推論されることを確認します。
- ルート定義から呼び出し側まで、ジェネリクスを手書きせずに型が貫通する感覚を掴みます。

### 確認手順

1. `src/routes/posts.$postId.tsx` を作成します。

   ```tsx
   export const Route = createFileRoute("/posts/$postId")({
     component: PostDetail,
   });

   function PostDetail() {
     const { postId } = Route.useParams(); // postId: string が推論される
     return <div>Post: {postId}</div>;
   }
   ```

2. `Route.useParams()` にホバーして、戻り値の型が `{ postId: string }` であることを確認します。
3. `<Link to="/posts/$postId" params={{ postId: '1' }} />` を書き、`params` のキーが要求されることを型レベルで確認します。

### Next.js 対比

| 観点          | TanStack Router                     | Next.js 15+ (App Router)                             |
| ------------- | ----------------------------------- | ---------------------------------------------------- |
| ファイル名    | `posts.$postId.tsx`                 | `app/posts/[postId]/page.tsx`                        |
| params の型   | `Route.useParams()` で自動推論      | `params: Promise<{ postId: string }>` を自分で型注釈 |
| 取得方法      | 同期的に取得可能                    | `await params` が必要                                |
| Link の型安全 | `to` と `params` がペアで検査される | `href` は文字列ベース                                |

> Next.js 15 以降の `params` が `Promise` 化された経緯と比較すると、TanStack Router が「ルート定義 = 型の真実」という思想で一貫していることが際立ちます。

## 実験 3: Layout Route で `<Outlet />` を理解する

### ファイル例

- `src/routes/posts.tsx` (layout)
- `src/routes/posts.index.tsx` (一覧)
- `src/routes/posts.$postId.tsx` (詳細, 実験 2 で作成済み)

### 狙い

- ドット区切りのフラットなファイル名で、ディレクトリ階層と同じレイアウト共有が実現できることを体感します。
- 親ルート (`posts.tsx`) に置いた `<Outlet />` の位置に子ルートが差し込まれる仕組みを理解します。

### 確認手順

1. `src/routes/posts.tsx` を作成し、共通ヘッダーやサイドナビと `<Outlet />` を配置します。

   ```tsx
   export const Route = createFileRoute("/posts")({
     component: PostsLayout,
   });

   function PostsLayout() {
     return (
       <section>
         <h1>Posts</h1>
         <Outlet />
       </section>
     );
   }
   ```

2. `src/routes/posts.index.tsx` を作成し、`/posts` にアクセスしたときに layout の中で一覧が表示されることを確認します。
3. `/posts/1` にアクセスし、同じ layout の中で詳細 (実験 2) が描画されることを確認します。

### Next.js 対比

| 観点            | TanStack Router                       | Next.js (App Router)                  |
| --------------- | ------------------------------------- | ------------------------------------- |
| layout ファイル | `posts.tsx` (ドット区切り、フラット)  | `app/posts/layout.tsx` (ディレクトリ) |
| index ページ    | `posts.index.tsx`                     | `app/posts/page.tsx`                  |
| 子の挿入箇所    | `<Outlet />`                          | `children` prop                       |
| 階層表現        | ファイル名で表現 (ディレクトリでも可) | ディレクトリ階層で表現                |

> TanStack Router は同じ概念を「フラットなファイル名」と「ディレクトリ階層」の両方で書ける点が特徴です。プロジェクトの規模や好みに応じて選べます。

## 実験 4: Pathless Layout (`_auth.tsx`) で URL に出ない layout

### ファイル例

- `src/routes/_auth.tsx`
- `src/routes/_auth.dashboard.tsx`

### 狙い

- `_` プレフィックスが付いた layout は URL セグメントに現れず、子ルートだけが URL に反映されることを確認します。
- 「ログイン必須のページ群を共通の認可ロジックでラップする」といったユースケースに直接マッピングできることを体感します。

### 確認手順

1. `src/routes/_auth.tsx` を作成し、`beforeLoad` で認可チェックを行い、`<Outlet />` を返すコンポーネントを定義します。
   ```tsx
   export const Route = createFileRoute("/_auth")({
     beforeLoad: ({ context }) => {
       // 未ログインなら redirect する想定
     },
     component: () => <Outlet />,
   });
   ```
2. `src/routes/_auth.dashboard.tsx` を作成します。URL は `/dashboard` になります (`_auth` は URL に出ない)。
3. ブラウザで `/dashboard` にアクセスし、`_auth` の `beforeLoad` を通過してから描画されることを確認します。

### Next.js 対比

| 観点         | TanStack Router (`_auth`)                                                | Next.js (`(auth)`)                             |
| ------------ | ------------------------------------------------------------------------ | ---------------------------------------------- |
| 記法         | アンダースコア prefix (`_auth.tsx`)                                      | 丸括弧グループ (`app/(auth)/...`)              |
| 役割         | **layout 単位の適用** (親に layout や `beforeLoad` を持てる)             | 単なる **グルーピング** (URL から除外するだけ) |
| 共有ロジック | `_auth.tsx` 自体が Route として `beforeLoad`/`loader`/`component` を持つ | `(auth)/layout.tsx` を別途用意して持たせる     |
| URL への影響 | `_auth` セグメントは出ない                                               | `(auth)` セグメントは出ない                    |

> Next.js の `(group)` は「URL からの除外」だけが目的ですが、TanStack Router の `_layout` は「layout として機能しつつ URL に出ない」という、より積極的な意味を持ちます。

## 体感の観点 (各実験共通でチェックする)

- **ファイル名のフラット性と階層性の両対応**: `posts.$postId.tsx` でも `posts/$postId.tsx` でも同じルートツリーが生成されることを、各実験で意識的に切り替えてみると理解が深まります。
- **`__root.tsx` の意味**: Next.js の `app/layout.tsx` (Root Layout) に相当しますが、TanStack Router では「ルートツリーの親」が明示的に 1 ファイルとして存在します。これにより、グローバルな `loader`/`context`/`<Outlet />` が型レベルで一意に決まります。
- **`routeTree.gen.ts` の自動生成**: 各実験でファイルを追加・リネーム・削除した際に `routeTree.gen.ts` がどう変化するかを観察します。これがファイルベースルーティングの「真実の源」であり、`<Link>` の型補完や `Route.useParams()` の型推論の根拠になっています。
